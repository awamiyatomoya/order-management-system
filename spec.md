# 受注管理システム MVP 仕様

## 目的

化粧品ブランドの卸売 B2B 受注業務で、使えなくなった STORiiY の受注管理部分を置き換える。MVP は「発注ファイル取り込み、商品マスタで正規化、受注DBに保存、クライアント別に閲覧」までに集中する。

## MVP で作るもの

- 社内オペレーター向けの管理画面
- クライアント選択後、そのクライアントの受注だけを表示・登録
- 卸先の CSV/Excel ファイル取り込み
- PDF 発注書の取り込み
- PDF 画像の OCR 読み取り
- 卸先ごとの列マッピング設定
- JAN をキーにした商品マスタ照合
- 未登録JANがある場合、その場で商品登録
- 登録済み商品の価格・商品名・COOOLa商品コード更新
- 正常ファイルは確認なしで自動保存
- 怪しいファイルは保存せず、理由を表示
- 保存直後の状態は `imported`（画面表示は `取込済み`）
- 担当者が確定ボタンを押すと `confirmed`（画面表示は `確定済み`）
- `confirmed` 時点の下代・税率・金額を注文明細に固定
- `confirmed` から `imported` へ取り消し可能
- `imported` / `confirmed` の受注削除
- 取込履歴とエラー履歴のDB保存・再表示
- 受注一覧の検索・ステータス絞り込み

## MVP で作らないもの

- クライアントログイン
- COOOLa の再実装
- COOOLa への API 連携
- ライブEDI受信サーバ
- 月次請求、セルイン可視化、納品書生成
- 出荷指示済み以降の業務フロー
- 権限管理、監査ログ

## 確定した運用ルール

- 受注データは必ず `client_id` を持つ
- 画面でクライアントを選ばないと操作できない
- 明細の結合キーは JAN
- COOOLa 商品コード、下代、商品名は商品マスタから補完する
- 確定時に単価・税率を明細にスナップショットする
- 同じ `supplier_id + order_no` の再取り込みは、既存受注が `imported` なら上書きする
- 既存受注が `confirmed` 以降なら再取り込みを止める
- 取込履歴とエラーはDBに残す前提で設計する
- 1つでもエラーがあればファイル全体を保存しない
- 未登録JANはその場で商品登録し、登録後に自動再チェックする
- 商品マスタ更新後、`imported` の受注金額は現在の商品マスタ価格で再計算される
- `confirmed` の受注金額は確定時スナップショットを使う
- 削除できるのは `imported` / `confirmed` の受注までとする
- 将来の `shipping_instructed` / `shipped` は削除不可とする

## 確認済みの主要フロー

1. PDF発注書をアップロードする
2. macOS OCRで発注書を読み取る
3. 発注番号、発注日、着荷指定日、JAN、数量を受注へ変換する
4. 未登録JANを検出する
5. 商品マスタへ商品を登録する
6. 商品価格を更新して受注金額を発注書と一致させる
7. 受注を `imported` として保存する
8. 受注を `confirmed` に確定する
9. 同じPDFの再取込が確定済みブロックされる
10. 受注を削除し、再取込できる
11. ブラウザ再読み込み後も受注と取込履歴がSupabaseから表示される

## 技術方針

- Next.js App Router + TypeScript
- Supabase クライアントでDBアクセスを行う
- サーバー側保存処理は Server Action を使う
- PDF解析は Route Handler を使う
- PDF内テキスト抽出には `pdf-parse` を使う
- 画像PDFのOCRには macOS Vision OCR を優先し、Tesseract OCR を補助として使う
- PDF画像化は macOS Quick Look と `pdf-parse` のレンダリングを併用する
- 本番データ、発注ファイル、秘密キー、`.env.local` は Git に入れない
- 仕様書、マッピング設定ひな形、匿名サンプル、テスト用ダミーデータは Git に入れてよい
