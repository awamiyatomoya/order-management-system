# 受注管理システム MVP 仕様

## 目的

化粧品ブランドの卸売 B2B 受注業務で、使えなくなった STORiiY の受注管理部分を置き換える。MVP は「発注ファイル取り込み、商品マスタで正規化、受注DBに保存、クライアント別に閲覧」までに集中する。

## MVP で作るもの

- 社内オペレーター向けの管理画面
- クライアント選択後、そのクライアントの受注だけを表示・登録
- 卸先の CSV/Excel ファイル取り込み
- 卸先ごとの列マッピング設定
- JAN をキーにした商品マスタ照合
- 未登録JANがある場合、その場で商品登録
- 正常ファイルは確認なしで自動保存
- 怪しいファイルは保存せず、理由を表示
- 保存直後の状態は `imported`
- 担当者が確定ボタンを押すと `confirmed`
- `confirmed` 時点の下代・税率・金額を注文明細に固定
- `confirmed` から `imported` へ取り消し可能

## MVP で作らないもの

- クライアントログイン
- COOOLa の再実装
- COOOLa への API 連携
- ライブEDI受信サーバ
- 月次請求、セルイン可視化、納品書生成

## 確定した運用ルール

- 受注データは必ず `client_id` を持つ
- 画面でクライアントを選ばないと操作できない
- 明細の結合キーは JAN
- COOOLa 商品コード、下代、商品名は商品マスタから補完する
- 確定時に単価・税率を明細にスナップショットする
- 同じ `supplier_id + order_no` の再取り込みは、既存受注が `imported` なら上書きする
- 既存受注が `confirmed` 以降なら再取り込みを止める
- 取込履歴とエラーはDBに残す前提で設計する
- 1つでもエラーがあればファイル全体を保存しない
- 未登録JANはその場で商品登録し、登録後に自動再チェックする

## 技術方針

- Next.js App Router + TypeScript
- Supabase クライアントに統一する予定
- MVP 初期は仮データとブラウザ内状態で画面・取込フローを先に作る
- 本番データ、発注ファイル、秘密キー、`.env.local` は Git に入れない
- 仕様書、マッピング設定ひな形、匿名サンプル、テスト用ダミーデータは Git に入れてよい
