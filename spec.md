# 受注管理システム 要件定義

## 1. 目的

化粧品ブランドの卸売 B2B 受注業務において、使えなくなった STORiiY の受注管理部分を置き換える。

MVP では、卸先から届く発注ファイルを取り込み、商品マスタで正規化し、受注として保存・確認・確定できる状態を作る。COOOLa そのものの再実装ではなく、将来的に COOOLa へ渡せる受注データを整えることを目的とする。

## 2. 対象ユーザー

- 社内オペレーター: 発注ファイルを取り込み、エラー確認、商品マスタ補完、受注確定を行う
- 管理者: 商品マスタや卸先マッピングを整備し、取込履歴を確認する
- クライアント担当者: 将来的に自社分の受注を確認する想定。ただし MVP ではログイン機能は作らない

## 3. 対象業務

MVP の対象は、発注書受領から受注確定までとする。

1. クライアントを選択する
2. 卸先を選択する
3. CSV / Excel / PDF の発注ファイルを取り込む
4. 発注番号、発注日、着荷指定日、JAN、数量を抽出する
5. JAN をキーに商品マスタと照合する
6. 未登録 JAN や不正データを確認する
7. 必要に応じて商品マスタを登録・更新する
8. 問題がなければ受注を `取込済み` として保存する
9. 金額や内容を確認し、受注を `確定済み` にする
10. 取込履歴・エラー履歴から過去の処理結果を確認する

## 4. MVP スコープ

### 作るもの

- 社内オペレーター向けの管理画面
- クライアント別の受注表示・登録
- 卸先別の CSV / Excel 取込
- PDF 発注書の取込
- PDF 画像の OCR 読み取り
- 卸先ごとの列マッピング設定
- JAN をキーにした商品マスタ照合
- 未登録 JAN の検出と商品登録
- 登録済み商品の価格・商品名・COOOLa 商品コード更新
- 受注の `取込済み` 保存
- 受注の `確定済み` への確定
- 確定取消
- `取込済み` / `確定済み` 受注の削除
- 確定済み受注の再取込ブロック
- 取込履歴とエラー履歴の保存・再表示
- 受注一覧の検索・ステータス絞り込み

### 作らないもの

- クライアントログイン
- COOOLa の再実装
- COOOLa への API 連携
- ライブ EDI 受信サーバ
- 月次請求
- セルイン可視化
- 納品書生成
- 出荷指示済み以降の業務フロー
- 権限管理
- 監査ログ

## 5. 機能要件

### 5.1 クライアント選択

- すべての受注データは `client_id` を必須で持つ
- 画面でクライアントを選択するまで、受注の取込・表示・操作はできない
- 受注一覧、商品、取込履歴、エラー履歴は選択中の `client_id` で絞り込む

### 5.2 発注ファイル取込

- CSV、Excel、PDF の発注ファイルを取り込める
- CSV / Excel は卸先ごとのマッピング設定に従って解析する
- PDF はテキスト抽出を行い、画像 PDF の場合は OCR を使う
- 解析結果から、少なくとも発注番号、発注日、着荷指定日、JAN、数量を抽出する
- 1つでも保存できないエラーがある場合、ファイル全体を保存しない
- エラー内容は画面に表示し、DB に履歴として保存する

### 5.3 商品マスタ照合

- 受注明細の結合キーは JAN とする
- COOOLa 商品コード、下代、商品名、税率は商品マスタから補完する
- 未登録 JAN がある場合は、画面上で商品登録できる
- 商品登録後、取込データを自動再チェックする
- 登録済み商品の価格、商品名、COOOLa 商品コードを更新できる

### 5.4 受注保存

- 正常に解析・照合できたファイルは受注として保存する
- 保存直後の状態は `imported` とし、画面では `取込済み` と表示する
- 同じ `supplier_id + order_no` の再取込は、既存受注が `imported` の場合のみ上書きできる
- 既存受注が `confirmed` 以降の場合、再取込はブロックする

### 5.5 受注確定

- 担当者が確認後、受注を `confirmed` に変更できる
- `confirmed` は画面では `確定済み` と表示する
- 確定時点の単価、税率、金額を受注明細にスナップショット保存する
- 商品マスタ更新後も、`confirmed` の受注金額は確定時スナップショットを使う
- `confirmed` から `imported` へ確定取消できる

### 5.6 受注削除

- `imported` / `confirmed` の受注は削除できる
- 将来追加される `shipping_instructed` / `shipped` の受注は削除不可とする
- 削除後は、同じ発注番号の再取込ができる

### 5.7 受注一覧・履歴

- 受注一覧を発注番号、PDF ファイル名、届け先、ステータスで検索・絞り込みできる
- 取込履歴を一覧で確認できる
- 取込エラー履歴を一覧で確認できる
- ブラウザ再読み込み後も、Supabase から受注と履歴を再表示できる

## 6. データ要件

主なテーブルは以下とする。

- `clients`: クライアント
- `suppliers`: 卸先
- `products`: 商品マスタ
- `orders`: 受注ヘッダ
- `order_lines`: 受注明細
- `import_batches`: 取込履歴
- `import_errors`: 取込エラー

すべての業務テーブルは `client_id` を持ち、クライアント間のデータ混在を防ぐ。

## 7. 重要な運用ルール

- 発注ファイルの取り込み前に、必ずクライアントと卸先を選択する
- 卸先ごとの CSV / Excel パーサーはコードに直書きせず、`config/suppliers/*.json` の設定で追加する
- 受注金額は、`imported` では現在の商品マスタ価格で再計算する
- 受注金額は、`confirmed` では確定時スナップショットを使う
- 取込履歴とエラー履歴は DB に残す
- 本番の発注ファイル、住所、電話番号、取引先名、秘密キー、`.env.local` は Git に入れない
- 匿名サンプル、仕様書、マッピング設定ひな形、テスト用ダミーデータは Git に入れてよい

## 8. 受け入れ条件

MVP は以下を満たしたら完了とする。

1. クライアントを選択し、そのクライアントの受注だけを表示できる
2. PDF 発注書をアップロードできる
3. macOS OCR または Tesseract OCR で発注書を読み取れる
4. 発注番号、発注日、着荷指定日、JAN、数量を受注へ変換できる
5. 未登録 JAN を検出できる
6. 未登録 JAN を商品マスタへ登録し、取込データを再チェックできる
7. 商品価格を更新し、受注金額を再計算できる
8. 問題ない受注を `取込済み` として保存できる
9. `取込済み` の受注を `確定済み` にできる
10. 確定済み受注の単価、税率、金額がスナップショットされる
11. 同じ発注書を再取込したとき、確定済み受注は上書きされない
12. 受注を削除し、同じ発注書を再取込できる
13. ブラウザ再読み込み後も、受注、取込履歴、エラー履歴が表示される
14. 受注一覧を検索・ステータス絞り込みできる

## 9. 技術方針

- Next.js App Router + TypeScript を使う
- Supabase を受注・商品・履歴データの保存先にする
- サーバー側保存処理は Server Action を使う
- PDF 解析は Route Handler を使う
- PDF 内テキスト抽出には `pdf-parse` を使う
- 画像 PDF の OCR には macOS Vision OCR を優先し、Tesseract OCR を補助として使う
- PDF 画像化は macOS Quick Look と `pdf-parse` のレンダリングを併用する
