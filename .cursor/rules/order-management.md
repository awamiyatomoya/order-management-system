# 受注管理システムの不変ルール

- すべてのテーブルに `client_id` を必須で持たせ、全クエリ・ビューを `client_id` で絞ること。クライアント間のデータ混在を絶対に起こさない。
- 受注明細の結合キーは JAN。COOOLa商品コード・下代・商品名は `products` テーブル参照で補完する。明細に直書きしない。
- 卸先ごとのCSV/Excelパーサーをコードに直書きしない。必ず設定ファイル `config/suppliers/*.json` で追加する。
- 受注確定時に単価・税率をスナップショットして明細に保存する。マスタ価格変更で過去受注の金額をブレさせない。
- COOOLa を再実装しない。連携はファイル出力で行う。発注はファイル取り込み前提。ライブEDIサーバは作らない。
- まずは MVP（取込、正規化、受注DB、クライアント別閲覧）の受け入れ条件だけを満たす。後続機能は実装しない。
- 本番の発注ファイル、住所、電話番号、取引先名、秘密キー、`.env.local` は Git に入れない。匿名サンプルだけをコミット対象にする。
